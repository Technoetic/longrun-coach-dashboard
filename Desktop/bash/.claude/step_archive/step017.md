# Step 17 - Awwwards UX/UI·레이아웃 통합 조사

## 실행 내용

전체 조사결과를 기반으로, Awwwards 수상작에서 UX/UI 패턴, 레이아웃, 인터랙션, 시각 디자인을 통합 조사한다. **이 조사가 이후 설계·구현의 유일한 디자인 기준이 된다.**

**필요한 파일:**

- step013_조사결과_chunk*.md (전체 조사결과)

**조사 방법:**

1. 전체 조사결과에서 프로젝트 유형, 주요 기능, 대상 사용자, UI 구성 요소를 추출한다.
2. 추출한 정보를 기반으로 Awwwards(https://www.awwwards.com/) 수상작에서 관련 사례를 조사한다.
3. **조사 축을 동적으로 생성**하여 UX/UI + 레이아웃을 통합 조사한다.
4. **각 Awwwards 수상작 사이트를 Playwright로 방문하여 스크린샷을 촬영한다.**

**스크린샷 규칙:**

- 각 참고 사이트의 **모든 페이지**를 Playwright로 방문하여 뷰포트별 스크린샷을 촬영한다.
- **페이지 수집 규칙:**
  - 랜딩 페이지뿐 아니라 네비게이션에서 접근 가능한 모든 페이지를 촬영한다
  - 각 페이지마다 뷰포트별로 촬영한다
  - **SPA 대응:** 네비게이션 링크가 1개 이하인 경우 SPA로 판단하고 다음을 수행한다:
    - 페이지 전체를 스크롤하며 **모든 섹션을 빠짐없이** 촬영한다 (섹션명: `section1`, `section2`, ...)
    - 뷰포트 높이 단위로 스크롤하며 페이지 끝까지 촬영한다 — 섹션 수를 제한하지 않는다
    - 버튼/메뉴 클릭 등 인터랙션으로 전환되는 화면이 있으면 전환 후 상태도 촬영한다 (상태명: `open`, `active`, `expanded` 등)
    - 예: `awwwards-stripe-section2-desktop.png`, `awwwards-stripe-menu-open-mobile.png`
- **뷰포트 결정 규칙:**
  - 프로젝트 특성을 분석하여 필요한 뷰포트를 동적으로 결정한다
  - 반응형 프로젝트: 데스크톱 (1920×1080) + 태블릿 (768×1024) + 모바일 (390×844)
  - 비반응형 프로젝트: 데스크톱 (1920×1080)만
- **파일명 규칙:** `awwwards-{사이트명}-{페이지명}-{뷰포트}.png`
  - 사이트명: 소문자, 하이픈 구분, 도메인명 기반
  - 페이지명: 소문자, 하이픈 구분 (예: `home`, `about`, `features`, `pricing`)
  - 뷰포트: `desktop`, `tablet`, `mobile`
  - 예: `awwwards-stripe-home-desktop.png`, `awwwards-stripe-pricing-mobile.png`
- 저장 경로: `.claude/screenshots/research/`
- 스크린샷 파일명 충돌 방지 (에이전트별 고유 prefix)
- **스크린샷과 원본 데이터가 없으면 조사 무효!**

**조사 축 생성 규칙:**

- 기획 결과의 기능/UI 요구사항을 분석하여 UX/UI·레이아웃에 영향을 주는 요소를 식별한다
- 식별된 요소마다 하나의 조사 축을 생성한다 (레이아웃 구조, 컨트롤 패널 배치, 색상/타이포그래피, 인터랙션, 반응형 등)
- 사전에 고정된 축 목록을 사용하지 않는다

**각 조사 축에서 수행할 내용:**

- Awwwards 수상작 중 해당 축과 관련된 참고 사례를 최소 3개 이상 수집·분석한다
- 각 사례의 레이아웃 패턴, UX/UI 패턴, 인터랙션, 시각적 구성을 기록한다
- 실현 가능한 대안을 최소 2개 이상 도출하고 장단점을 비교한다
- 특정 대안을 선택하지 않는다 — 비교 자료만 제공하고 결정은 기획 단계에서 한다

**조사 원칙:**

- 특정 기술/패턴/구조를 사전에 결정하거나 권장하지 않는다
- 기획 결과에 명시되지 않은 요소를 임의로 추가하지 않는다
- 장점과 단점을 균형 있게 기술한다

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용해야 한다.

**조사결과는 청크 단위로 저장한다:**

```
step017_조사결과_chunk1.md (500줄 이하)
step017_조사결과_chunk2.md (500줄 이하)
step017_조사결과_chunk3.md (500줄 이하)
...
```

**작성 규칙**:

- 각 청크는 500줄 이하로 작성 (성능 최적화)
- `.claude/hooks/research-validator.ps1`에서 각 청크 검증 (BOM/CRLF/줄수/파일크기)
- 청크 그대로 유지 (병합 안 함)

서브에이전트는 항상 haiku를 사용한다.
